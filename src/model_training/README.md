# Task C: Model Training & Preference Alignment

Owns the SFT (Supervised Fine-Tuning) and DPO (Direct Preference Optimization) training pipelines. This module is the final evolutionary step of the project, responsible for converting an untuned base LLM into a highly constrained, citation-aware **Fact-Checking Agent**.

---

## What is ready

| Deliverable | Location | Notes |
|-------------|----------|-------|
| SFT Adapter | `models/fact_checker_sft/` | LoRA weights trained to understand parsing evidence and outputting Verdicts. |
| DPO Adapter | `models/fact_checker_dpo/` | Final LoRA weights mathematically optimized to penalize hallucinations. |
| SFT Script  | `src/model_training/train_sft.py` | Standalone SFT trainer using `SFTConfig` and 4-Bit options. |
| DPO Script  | `src/model_training/train_dpo.py` | Standalone DPO trainer to align preferences. |

---

### Quick Start (Model Training)

If you have downloaded the datasets generated by **Person A (Data & Ingestion)**, you are ready to train the local `Qwen2.5-0.5B` baseline model.

```bash
# 1. Run Supervised Fine-Tuning (SFT) over the dataset for 3 full epochs
python -m src.model_training.train_sft --epochs 3 --max_samples 150000

# 2. Align the model against hallucination using Direct Preference Optimization
#    Note: This explicitly stacks onto the weights generated by Step 1
python -m src.model_training.train_dpo --epochs 1 --max_samples 150000
```
>*Note: If you run this without a CUDA-capable machine, `--max_samples` should be restricted heavily or the pipeline will take an eternity to execute.*

### Command-Line Arguments
Both `train_sft` and `train_dpo` share the same interface for scaling from laptop debugging to full server production.

| Argument | Default | Description |
|-----------|---------|-------------|
| `--epochs` | `1` | Total number of full passes through the training dataset. (Recommended: 3 for SFT, 1 for DPO). |
| `--max_samples` | *Varies* | Slices the `data/processed/*.jsonl` rows to save RAM. If set to a large number (e.g. `200000`), it uses the full dataset. |
| `--max_steps` | `None` | Overrides epochs to instantly kill the script after exactly `X` learning steps. Perfect for 10-second PyTorch smoke tests. |

### Validating the Model (Inference / Pytest)

Once you have generated the `./models/fact_checker_dpo/` adapters, you can automatically mount them to test the intelligence of the model!

```bash
# Run the Pytest evaluation suite to aggressively test Abstention logic
pytest tests/test_model_inference.py -v -s
```

## Technical Details

### The Data Prompts (`data_prep.py`)
To prevent data contamination, all formats leverage native `apply_chat_template` matching exactly what the model expects. 

The prompt strictly enforces:
1. `[source_id]` annotations next to all incoming evidence.
2. The user passes `Claim: [text] Evidence: [list]`.
3. The Assistant responds systematically with `Verdict: [SUPPORTED/REFUTED/NOT_ENOUGH_INFO] \n Explanation: [...]`.

### DPO Hallucination Penalization
The magic of this module lies in `data_prep.py`'s `prepare_dpo_dataset()`. 
If an incoming claim lacks supportive evidence (`NOT_ENOUGH_INFO`), the pipeline spins up a **Synthetic Hallucination**. 
* **Chosen (Reward):** The model safely abstains and says there is no info.
* **Rejected (Penalty):** A synthetic completion where the model claims it is `SUPPORTED` purely out of thin air.

When `train_dpo.py` runs, we map log-probability math against those synthetic pairs, effectively rewiring the model's brain to view "Confident Guessing" as a mathematically incorrect gradient!
